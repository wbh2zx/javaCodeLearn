## 一、Docker 基础与核心操作（企业级部署必备）

- **子技能1：Docker 核心概念与基础命令（镜像/容器/仓库）**
        最主流写法（2026使用率 100% 企业项目）。为什么主流：Docker 核心基石，所有容器化部署的基础，核心命令（docker build/run/ps/stop/rm/images）是日常开发、运维的必备操作，适配所有企业级项目（Spring Boot微服务、中间件、前端项目），2026年主流Docker版本为26.x。
      

- **子技能2：Dockerfile 基础编写（多阶段构建/基础镜像选型）**
        适用场景：所有容器化项目（自定义镜像、项目打包部署）。优缺点：基础镜像选型（如alpine、openjdk:slim）可大幅缩减镜像体积，多阶段构建可排除无用依赖，提升部署效率；缺点是基础镜像选型不当易导致依赖缺失，多阶段构建步骤写错会增加镜像体积。
      

- **隐藏/冷门但生产中踩过的坑**：① 基础镜像选择过新/过旧，导致项目依赖不兼容（如openjdk:8-jdk-alpine缺失系统库）；② Dockerfile 中未清理构建缓存（如apt/yum缓存），导致镜像体积暴涨（从几百MB增至几GB）；③ 容器启动时未指定健康检查，导致服务异常后无法自动重启；④ 挂载目录权限配置错误（如宿主机目录权限不足），导致容器无法读写文件。真实案例：2025年某Spring Boot项目Dockerfile未使用多阶段构建，镜像体积达2.3GB，部署时网络超时，导致上线延迟1小时。
      

- **面试/私活高频考点**：Docker 核心概念（镜像、容器、仓库）的区别、Dockerfile 核心指令（FROM/COPY/RUN/ENTRYPOINT/EXPOSE）用法、多阶段构建的优势、基础镜像选型技巧、Docker 镜像分层原理。

## 二、Docker 进阶配置与优化（生产级性能提升）

- **子技能1：Docker 镜像优化（瘦身/缓存/加速）**
        最主流写法（2026使用率 95%+ 企业项目）。为什么主流：镜像体积直接影响部署速度、磁盘占用，优化后可提升部署效率、降低运维成本，核心优化手段（多阶段构建、基础镜像瘦身、清理缓存、镜像分层复用）是企业级容器化的必备能力。
      

- **子技能2：Docker 容器网络配置（桥接/host/自定义网络）**
        适用场景：多容器通信（如Spring Boot服务+MySQL+Redis）、微服务容器化部署。优缺点：自定义网络可隔离容器、避免端口冲突，host网络可提升网络性能；缺点是host网络会暴露容器端口，存在安全风险，自定义网络配置复杂易出错。
      

- **隐藏/冷门但生产中踩过的坑**：① 镜像分层过多（Dockerfile 中RUN指令过多），导致镜像构建缓慢、体积增大；② 未配置Docker镜像加速器（如阿里云、网易加速器），导致镜像拉取速度极慢；③ 自定义网络未配置网关，导致多容器之间无法通信；④ 容器端口映射冲突，导致服务启动失败，排查成本高。真实案例：某微服务项目容器使用host网络，未隔离端口，导致不同服务端口冲突，服务启动异常，排查耗时3小时。
      

- **面试/私活高频考点**：Docker 镜像优化的核心方法、Docker 网络模式（bridge/host/overlay/none）的区别及适用场景、容器端口映射（-p/-P）的用法、镜像加速器配置、容器间通信的实现方式。

## 三、Docker Compose 实战（单机/小规模集群部署）

- **子技能1：Docker Compose 核心配置与一键部署**
        最主流写法（2026使用率 90%+ 中小规模项目）。为什么主流：简化多容器部署流程，通过docker-compose.yml配置文件，一键启动/停止所有关联容器（如Spring Boot服务、数据库、Redis、Nacos），适配单机部署、测试环境、小规模生产集群，与CI/CD流水线无缝集成。
      

- **子技能2：Docker Compose 进阶配置（依赖顺序/环境变量/挂载）**
        适用场景：复杂多容器项目（微服务单机部署、前后端分离项目）。优缺点：可配置容器依赖顺序、环境变量、数据卷挂载，灵活适配不同环境；缺点是不支持大规模集群部署，配置文件复杂易出错，容器扩缩容不便。
      

- **隐藏/冷门但生产中踩过的坑**：① 未配置容器依赖顺序（如Spring Boot服务先于MySQL启动），导致服务连接数据库失败；② 环境变量配置错误（如数据库地址、密码写错），导致容器启动后功能失效；③ 数据卷挂载路径写错，导致数据丢失（如宿主机路径不存在）；④ Docker Compose 版本与Docker版本不兼容，导致配置文件无法解析。真实案例：某项目docker-compose.yml未配置depends_on，Spring Boot服务先于MySQL启动，导致服务启动失败，线上故障持续20分钟。
      

- **面试/私活高频考点**：Docker Compose 核心配置（version/services/volumes/networks）用法、容器依赖顺序配置（depends_on）、环境变量注入方式、数据卷挂载原理、Docker Compose 与 Docker 命令的区别。

## 四、Docker 数据管理与安全（生产级可靠性保障）

- **子技能1：Docker 数据卷（Volume）与绑定挂载（Bind Mount）**
        最主流写法（2026使用率 95%+ 企业项目）。为什么主流：解决容器数据持久化问题，避免容器删除后数据丢失，适配数据库、日志存储等场景，数据卷可实现多容器数据共享，绑定挂载可直接映射宿主机文件，灵活度高。
      

- **子技能2：Docker 容器安全（镜像安全/权限控制/镜像签名）**
        适用场景：生产环境容器部署、敏感业务（如支付、用户中心）。优缺点：镜像安全扫描可排查恶意镜像，权限控制可避免容器越权，镜像签名可保证镜像完整性；缺点是增加运维成本，配置复杂，部分安全策略会影响容器性能。
      

- **隐藏/冷门但生产中踩过的坑**：① 未使用数据卷，容器删除后数据丢失（如MySQL容器删除导致数据库数据丢失）；② 绑定挂载路径权限不足，导致容器无法读写数据；③ 使用非官方镜像，存在恶意代码、漏洞风险；④ 容器以root用户运行，存在越权风险，被黑客攻击后可控制宿主机。真实案例：某项目MySQL容器未使用数据卷，误删容器后数据库数据丢失，恢复数据耗时4小时，造成业务中断。
      

- **面试/私活高频考点**：Docker 数据卷（Volume）与绑定挂载（Bind Mount）的区别及适用场景、数据卷备份与恢复方法、Docker 镜像安全扫描工具（如Trivy）的使用、容器权限控制（非root用户运行）的实现、镜像签名与验证流程。

## 五、Docker 与 CI/CD 集成（企业级自动化部署）

- **子技能1：Docker + GitLab CI/Jenkins 实现自动化构建部署**
        最主流写法（2026使用率 90%+ 微服务项目）。为什么主流：实现从代码提交到镜像构建、部署的全流程自动化，减少人工操作，提升部署效率，降低人为错误，适配企业级微服务持续集成、持续部署需求，与Spring Boot项目无缝衔接。
      

- **子技能2：Docker 镜像仓库管理（Harbor/阿里云镜像仓库）**适用场景：企业级镜像存储、多团队协作、镜像版本管理。优缺点：私有镜像仓库可保证镜像安全，支持镜像版本控制、权限管理；缺点是Harbor部署复杂，需维护仓库服务器，增加运维成本。
      

- **隐藏/冷门但生产中踩过的坑**：① CI/CD流水线中，Docker镜像构建后未推送至私有仓库，导致部署时镜像拉取失败；② 镜像版本号管理混乱（如使用latest标签），导致部署版本错误，回滚困难；③ Harbor仓库权限配置不当，导致未授权用户可拉取/推送镜像；④ CI/CD流水线中，未做镜像安全扫描，恶意镜像被部署到生产环境。真实案例：某项目CI/CD流水线使用latest标签，误推送测试环境镜像至生产，导致生产服务异常，回滚耗时30分钟。
      

- **面试/私活高频考点**：Docker 与 GitLab CI/Jenkins 的集成步骤、镜像仓库（Harbor）的核心功能、镜像版本号管理最佳实践、CI/CD流水线中镜像构建与推送的脚本编写、镜像安全扫描在CI/CD中的集成。

## 六、Docker 与云原生/K8s 整合（2026企业级标配）

- **子技能1：Docker 镜像适配 K8s 部署（镜像规范/健康检查/资源限制）**
        最主流写法（2026使用率 95%+ 大规模微服务项目）。为什么主流：K8s 是大规模微服务集群的首选编排工具，而Docker是K8s的核心容器运行时（虽然有containerd替代，但Docker仍占主流），镜像规范、健康检查、资源限制是K8s部署的基础。


- **子技能2：Docker 与 containerd 选型与迁移**
        2026年新趋势。适用场景：大规模K8s集群、云原生部署。优缺点：containerd 轻量、性能更优，是K8s默认容器运行时；Docker 生态完善、易用性高，但冗余功能多；缺点是Docker与K8s的集成逐渐弱化，未来可能被containerd完全替代。
      

- **隐藏/冷门但生产中踩过的坑**：① Docker 镜像未配置健康检查，导致K8s误判容器健康状态，频繁重启容器；② 镜像未设置资源限制（CPU/内存），导致容器占用过多资源，影响其他服务；③ Docker 与 containerd 镜像格式不兼容，迁移时镜像无法运行；④ K8s 中容器使用root用户运行，被K8s安全策略拦截，无法启动。真实案例：某K8s集群部署Spring Boot容器，未配置资源限制，高并发下容器占用过多CPU，导致整个集群服务响应超时。
      

- **面试/私活高频考点**：Docker 与 K8s 的关系、Docker 镜像适配 K8s 部署的规范、健康检查在K8s中的作用、Docker 与 containerd 的区别及选型依据、容器资源限制（CPU/内存）的配置方法。

## 七、争议/边缘技能（架构师加分项）

- **子技能1：Docker vs Podman 选型**
        2026年争议点。适用场景：Podman 无守护进程、更安全，适配企业级安全需求；Docker 生态完善、易用性高，适配中小规模项目。优缺点：Podman 无需root用户、无守护进程漏洞，安全性更优；Docker 社区成熟、文档丰富，但存在守护进程安全风险。
      

- **子技能2：Docker Serverless 适配（如AWS ECS Fargate、阿里云FC）**
        冷门但未来趋势。适用场景：突发流量、按需付费场景（如活动营销服务）。优缺点：无需关注服务器部署，按需扩容，降低运维成本；缺点是冷启动延迟、调试困难，容器自定义配置受限，部分Docker特性适配不足。
      

- **隐藏/冷门但生产中踩过的坑**：① 盲目迁移Podman，导致现有Docker Compose配置无法使用，开发成本激增；② Serverless 容器冷启动延迟过高，导致活动峰值服务响应超时；③ 无服务架构下，容器日志收集、链路追踪困难，排查问题成本高。真实案例：某营销项目迁移Podman，因Docker Compose配置不兼容，修改配置耗时1周，未达到预期安全提升效果。
      

- **面试/私活高频考点**：Docker vs Podman 选型依据、Serverless 容器与传统Docker容器的区别、Docker 在云原生时代的发展趋势、容器化部署与Serverless部署的选型。
